# v0.5.0 — Inline language switching: design decisions log

Sesión de autopilot. Decisiones tomadas mientras el usuario estaba fuera.
Cada entrada es un punto donde tuve que elegir sin poder consultar.

## Visión

Cuando un párrafo en español incluye términos técnicos en inglés (`FastAPI`,
`Docker`, `Kubernetes`, etc.), pronunciarlos con voz inglesa para que suenen
correctamente, en lugar de fonetizarlos en español.

## Decisiones cerradas por el usuario antes de salir

| Tema | Decisión |
| --- | --- |
| Número de PRs | 1 sola PR con todas las fases |
| Default | Activado por defecto (breaking change → v0.5.0) |
| Release | Un solo tag `v0.5.0` al final, publish a PyPI tras merge |

| Enfoque | Híbrido (backticks=EN + heurística conservadora) |

## Fases

1. **Parser con Spans**: nuevo dataclass `Span(text, lang)`, campo aditivo en `Block`.
2. **Reader/Exporter**: consumen spans → múltiples llamadas a `edge_tts.Communicate` → concat MP3 bytes.
3. **CLI flags**: `--inline-code-lang en|none`, `--voice-en`, `--voice-es`.
4. **Heurística**: lista corta de términos técnicos EN para detectar fuera de backticks.
5. **Caché**: spans cortos repetidos cacheados en disco para acelerar exports.

## Decisiones tomadas en ruta

### 2026-XX: Descope de heurística (Phase 4) y caché en disco (Phase 5)

Tras completar Phases 1–3 (parser → reader/exporter → CLI flags) con 63/63
tests verdes, decido cerrar v0.5.0 sin la heurística de términos EN
fuera de backticks ni la caché en disco. Razones:

- **Heurística (frozenset ~200 términos)**: alto riesgo de falsos
  positivos en palabras ambiguas (`no`, `pin`, `mar`, `id`, `son`,
  `real`, `data` aparece en español puro como derivado de "dar"). Sin
  control fino del usuario para deshabilitar palabras concretas, una
  lista hardcoded acabaría más generando frustración que aportando.
  Mejor confiar en la convención existente: el autor de un `.md`
  técnico ya usa backticks alrededor de términos como `framework`,
  `pipeline`, `Docker`. Eso cubre el 90% del caso de uso con falsos
  positivos casi cero.
- **Caché en disco**: optimización prematura. La fast-path "todos los
  spans resuelven a la misma voz → una sola Communicate" ya elimina
  el coste extra en el caso común. Spans realmente multi-voz son una
  minoría del documento (típicamente <10% de los párrafos). No
  justifica meter complejidad de invalidación, paths por SO, sha256.

Ambas se quedan documentadas como follow-ups potenciales para v0.6.0
si el feedback real lo justifica.

### Default activado (breaking change menor)

`--inline-code-lang en` es el default. Documentos puramente en inglés
no cambian de comportamiento (todos los spans → voz EN). Documentos
en español sin backticks tampoco cambian (single-voice fast-path).
Solo cambia el comportamiento para documentos ES con backticks: el
contenido entre backticks ahora se pronuncia en inglés. El opt-out
es `--inline-code-lang none`.

### Fast-path para single-voice spans

Cuando todos los spans de un bloque resuelven a la misma voz (caso
común en docs monolingües), tanto `EdgeReader.play` como
`exporter._export_async` colapsan a una sola llamada a
`edge_tts.Communicate`. Esto evita el coste de N round-trips HTTPS
para los documentos que no necesitan multi-voz. Solo cuando hay un
salto real de voz se ejecuta la concatenación de bytes MP3.

### `forced_voice` desactiva multi-voz

Cuando el usuario pasa `--voice`, todos los spans se sintetizan con
esa voz. Tiene sentido: si fuerzas una voz concreta, no quieres que
el sistema decida por ti.
